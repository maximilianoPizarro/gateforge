Core Capabilities

Organizations running Red Hat 3scale API Management need a clear, automated path to Kubernetes-native API management via Red Hat Connectivity Link (Kuadrant). GateForge automates this migration end-to-end.

Discovery

Automatic 3scale Discovery

Discovers Products, Backends, Mapping Rules, and ActiveDocs via both CRD inspection and Admin API. Supports connecting to multiple 3scale instances simultaneously.

AI-Powered

AI Resource Generation & Verification

LangChain4j-based AI agent generates equivalent Connectivity Link resources, then reviews correctness, identifies potential issues, and provides recommendations.

Multi-Cluster

Multi-Cluster Deployment

Apply migrations to any target OpenShift cluster with ArgoCD auto-discovery and per-cluster RBAC validation. Download YAML for GitOps workflows.

Revert

Safe Rollback to 3scale

One-click revert of individual plans or bulk revert to restore 3scale management. Full audit trail with before/after YAML snapshots for every operation.

NEW v0.1.9

APICast Discovery & Migration

Discovers APIManager CRDs cluster-wide to find self-managed and multi-tenant APICast instances. Analyzes staging/production specs, custom Lua policies, TLS, and OpenTracing configuration. Maps APICast to Istio Gateway, EnvoyFilter, DestinationRule, and TelemetryPolicy.

Advanced Features
Hub-Spoke

Centralized Control Plane

PostgreSQL-backed persistence with Flyway migrations. Federated audit log across clusters. Hub overview API with aggregated stats and topology graph.

GitOps

Helm Chart & ArgoCD Ready

Packaged as a Helm chart on Artifact Hub. Deploys backend, frontend, and PostgreSQL with least-privilege RBAC. ArgoCD-native with selfHeal support.

Developer Hub

Backstage Integration

Auto-discovers 3scale products. Generates Component entities with providesApis. Dynamic plugin with Observability, Topology, and Editor tabs.

kuadrantctl

kuadrantctl-Powered Generation

Generates HTTPRoute, AuthPolicy, and RateLimitPolicy from OpenAPI specs. Auto-consolidates rules exceeding the Gateway API 16-rule limit into prefix-based matches.

CL 1.3

Connectivity Link 1.3 CRDs

Generates APIProduct, PlanPolicy, and APIKey CRDs from 3scale Application Plans, credentials, and product metadata.

Performance

FAQ Cache with Data Grid

Common AI questions cached in Red Hat Data Grid for instant responses. Observability via OpenTelemetry, Prometheus, and Grafana.

Install GateForge on OpenShift with PostgreSQL persistence.

# Add repo
helm repo add gateforge \
  https://maximilianopizarro.github.io/gateforge/

# Install with persistence
helm install gateforge gateforge/gateforge \
  --set ai.apiKey=YOUR_KEY \
  --set threescale.adminApi.url=URL \
  --set threescale.adminApi.accessToken=TOKEN

Developer Hub Integration

GateForge integrates with Red Hat Developer Hub (RHDH) 1.9+ through two OCI-packaged dynamic plugins and standard Software Templates. This section explains how to configure every piece.

Overview

After a successful migration, the GateForge backend triggers a Software Template (gateforge-register-component) via the RHDH Scaffolder API. The Scaffolder fetches the generated catalog-info.yaml from the GateForge backend and registers a Component entity in the catalog. Two dynamic plugins add frontend tabs (Observability, Policy Topology, Component Editor) and a backend processor for annotation-based entity enrichment.

1. GateForge Helm values

Set these values in your GateForge Helm chart deployment:

developerHub:
  enabled: true
  url: "https://backstage-developer-hub-<namespace>.<clusterDomain>"
  scaffolderUrl: "https://backstage-developer-hub-<namespace>.<clusterDomain>/api/scaffolder/v2/tasks"
  scaffolderToken: "<RHDH BACKEND_SECRET value>"
  componentSuffix: "-product"
ValueDescription
developerHub.scaffolderUrlFull URL to the Scaffolder task creation endpoint. Must match the RHDH external Route.
developerHub.scaffolderTokenThe BACKEND_SECRET value from the RHDH deployment. Used as Bearer token for Scaffolder API auth.
developerHub.componentSuffixSuffix added to the 3scale product name when creating Component entities (default: -product).

2. RHDH app-config-rhdh ConfigMap

Add the GateForge frontend Route to backend.reading.allow so the Scaffolder can fetch catalog-info.yaml:

backend:
  reading:
   allow:
    - host: github.com
    - host: raw.githubusercontent.com
    - host: gateforge-gateforge.<clusterDomain>

Register the Software Templates catalog in catalog.locations:

catalog:
  locations:
   - type: url
    target: https://github.com/your-org/repo/blob/main/software-templates/templates-catalog.yaml
  rules:
   - allow: [Component, Group, User, Resource, Location, Template, API]

3. Dynamic plugin registration (dynamic-plugins-rhdh)

Add both GateForge plugins to the dynamic-plugins-rhdh ConfigMap:

# Backend plugin (entity processor)
- package: oci://quay.io/maximilianopizarro/gateforge-devhub-plugin:0.1.9!gateforge-backstage-plugin-catalog-backend-module-gateforge
  disabled: false

# Frontend plugin (Observability, Topology, Editor tabs)
- package: oci://quay.io/maximilianopizarro/gateforge-devhub-frontend-plugin:0.1.9!gateforge-backstage-plugin-gateforge-frontend
  disabled: false
  pluginConfig:
   dynamicPlugins:
    frontend:
     gateforge.backstage-plugin-gateforge-frontend:
      mountPoints:
       - mountPoint: entity.page.observability/cards
        importName: GateForgeObservabilityTab
       - mountPoint: entity.page.topology/cards
        importName: GateForgeTopologyTab
       - mountPoint: entity.page.component/cards
        importName: GateForgeComponentEditorTab

4. RBAC permissions

Ensure users have the following permissions in rbac-policy-rhdh:

# Catalog — create locations and entities
p, role:default/authenticated, catalog.location.create, create, allow
p, role:default/authenticated, catalog.entity.create, create, allow

# Scaffolder — trigger templates
p, role:default/authenticated, scaffolder-template, read, allow
p, role:default/authenticated, scaffolder.task.read, read, allow
p, role:default/authenticated, scaffolder.task.create, create, allow
p, role:default/authenticated, scaffolder.action.execute, use, allow

5. Software Templates

Two templates are included in the GateForge repository:

TemplatePurposeTrigger
gateforge-register-componentRegisters a Component in the catalog using catalog:register action. Fetches catalog-info.yaml from the GateForge backend.Automatic (backend calls Scaffolder API after migration apply)
gateforge-unregister-componentRemoves a Component from the catalog using catalog:unregister action.Automatic (backend calls Scaffolder API on plan revert)

ArgoCD Deployment

GateForge can be deployed through ArgoCD using the Helm chart from the GitHub Pages repository. Two deployment topologies are supported.

Single-Cluster (Standalone)

The simplest setup: GateForge and all target resources run on the same OpenShift cluster. Create a single ArgoCD Application pointing at the Helm chart.

# ArgoCD Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: gateforge
  namespace: openshift-gitops
spec:
  project: default
  source:
   repoURL: https://maximilianopizarro.github.io/gateforge/
   chart: gateforge
   targetRevision: "0.1.9"
   helm:
    values: |
     clusterDomain: apps.cluster.example.com
     ai:
      apiKey: YOUR_KEY
      endpoint: https://your-llm-endpoint/v1
     threescale:
      adminApi:
       url: https://3scale-admin.example.com
       accessToken: TOKEN
     developerHub:
      enabled: true
      scaffolderUrl: https://backstage-developer-hub-dev-hub.apps.example.com/api/scaffolder/v2/tasks
      scaffolderToken: BACKEND_SECRET_VALUE
  destination:
   server: https://kubernetes.default.svc
   namespace: gateforge
  syncPolicy:
   automated:
    selfHeal: true
    prune: true

Hub-Spoke (Multi-Cluster)

In a hub-spoke topology, GateForge runs on the hub cluster and migrates APIs to one or more spoke clusters. The hub holds the PostgreSQL database, Data Grid cache, and serves the UI. Spoke clusters only need Connectivity Link (Kuadrant) installed.

# Hub cluster — GateForge with multi-cluster config
clusterDomain: apps.hub.example.com
argocd:
  clusterDiscovery: true

# OR explicit target clusters:
targetClusters: |
  [
   {
    "id": "spoke-east",
    "label": "East Region",
    "apiServerUrl": "https://api.spoke-east.example.com:6443",
    "token": "sa-token-for-spoke",
    "authType": "token",
    "verifySsl": false,
    "enabled": true
   },
   {
    "id": "spoke-west",
    "label": "West Region",
    "apiServerUrl": "https://api.spoke-west.example.com:6443",
    "token": "sa-token-for-spoke",
    "authType": "token",
    "verifySsl": false,
    "enabled": true
   }
  ]

ArgoCD tips

  • selfHeal: true ensures GateForge resources stay in sync with the chart.
  • Use ignoreDifferences for fields managed by OpenShift (e.g. Route status, Service cluster IPs).
  • To force a re-sync after updating the Helm chart at the same version, annotate the Application: argocd.argoproj.io/refresh=hard.
  • With argocd.clusterDiscovery: true, GateForge reads cluster secrets from the openshift-gitops namespace. The ServiceAccount needs get/list on Secrets in that namespace.

Installation Guide

Deploy GateForge on OpenShift or run it locally for development. Ensure prerequisites are satisfied before installing the Helm chart.

Prerequisites

  • OpenShift Container Platform 4.20+
  • Kuadrant Operator (Red Hat Connectivity Link)
  • 3scale Operator (for CRD-based discovery, where used)
  • Helm 3
  • kuadrantctl 0.10.x (bundled or installed where generation runs)

# Add the GateForge chart repository
helm repo add gateforge https://maximilianopizarro.github.io/gateforge/

# Install with required values

podman-compose up -d --build

Verify the deployment

  • Frontend UI: served on port 4200 when running the dev stack locally

Configuration: Multi-Tenant

Multi-tenant (hub-spoke) deployments connect GateForge to multiple 3scale sources and multiple target clusters. The hub cluster runs GateForge with PostgreSQL persistence and Data Grid; audit and plan data can be federated across spokes.

Multiple 3scale sources

Provide threescale.sources as a JSON array of source objects (id, label, admin URL, token, enabled flags). Each entry is discovered and listed independently in the UI and APIs.

Multiple target clusters

Set targetClusters to a JSON array of cluster definitions (API server URL, token, labels, TLS options). Migrations can target any registered cluster from the wizard.

Argo CD cluster discovery

With argocd.clusterDiscovery: true, GateForge discovers Argo CD cluster secrets and surfaces them as selectable targets without hard-coding kubeconfigs in values.

Persistence and audit

Keep postgresql.enabled: true on the hub for durable migration plans. Federated audit APIs aggregate events across clusters for compliance and troubleshooting.

AI Model Configuration

GateForge talks to any OpenAI-compatible endpoint through LiteLLM (or a compatible proxy). The default chart model name (deepseek-r1-distill-qwen-14b) is a sensible starting point but is not mandatory—point ai.endpoint and ai.model at whatever serving stack you run.

Configuration properties

PropertyDescription
ai.endpointBase URL for chat completions (OpenAI-compatible)
ai.modelModel id as understood by the upstream server
ai.apiKeyBearer token or API key required by the endpoint
ai.temperatureSampling temperature for completions
ai.maxTokensUpper bound on generated tokens per response

AI-Powered Analysis Flow

The assistant uses context injection plus a single chat completion (and tool calls where enabled). It is not a RAG pipeline: there are no vector stores or embedding indexes in the request path for these answers.

Chat request path

flowchart LR
  A["User POST /api/chat"] --> B["Normalize input"]
  B --> C{"FAQ cache hit?\n(Data Grid)"}
  C -->|Yes| R["Return cached answer"]
  C -->|No| D["Build context:\n3scale products,\ncluster status,\nkuadrantctl version"]
  D --> E["Inject context\ninto user message"]
  E --> F["LLM chat completion\n(OpenAI-compatible)"]
  F --> G["Strip <redacted_thinking> blocks\nfrom reasoning models"]
  G --> H["Response to client"]

  style C fill:#e3f2fd,stroke:#1565c0
  style F fill:#fff3e0,stroke:#e65100

Design decisions

  • No vector store or embeddings for this chat path—context is assembled from live services and cached key-value data.
  • FAQ warmup: 10 pre-defined prompts are sent at startup; answers are stored in Data Grid with a 24-hour TTL for fast repeated questions.
  • MigrationAgent system prompt includes concrete mapping examples from 3scale to Connectivity Link.
  • AI verification runs after YAML / resource generation to review the plan.
  • Reasoning models may emit <redacted_thinking>...</redacted_thinking> blocks; these are stripped before the user sees the answer.

Important notes

  • Only HTTPRoute generation via kuadrantctl is on the main migration path; AuthPolicy and RateLimitPolicy are typically built in Java for consistency with your cluster defaults.
  • Gateway API limits HTTPRoute matches to 16 rules; GateForge consolidates excess rules into prefix-style matches when needed.
  • If kuadrantctl exits with an error, the backend falls back to improved Java-based generation so migrations can continue.
