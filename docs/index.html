GateForge

EXPERIMENTAL This project is in early experimental phase. APIs and features may change without notice.

AI-powered migration platform for transitioning from Red Hat 3scale API Management to Red Hat Connectivity Link (Kuadrant) on OpenShift. GateForge automates discovery, analysis, resource generation, and validation — powered by LangChain4j, Model Context Protocol (MCP), and kuadrantctl.

What does GateForge solve?

Organizations running Red Hat 3scale API Management need a clear, automated path to Kubernetes-native API management via Red Hat Connectivity Link (Kuadrant). Manual migration is complex: mapping 3scale Products, Backends, Mapping Rules, Application Plans, and authentication configs to Gateway API HTTPRoutes, AuthPolicies, and RateLimitPolicies requires deep expertise in both ecosystems.

GateForge bridges this gap by:

Discovery

Automatically discovers 3scale Products, Backends, and ActiveDocs from both CRDs and the Admin API.

Analysis

AI-assisted analysis of your 3scale configuration to generate equivalent Connectivity Link resources.

Migration

Generates Gateway, HTTPRoute, AuthPolicy, and RateLimitPolicy YAML with one-click apply to cluster.

Validation

Post-migration test commands, audit trail, and bulk revert to 3scale if needed.

Installation

Install GateForge using Helm on OpenShift or run locally with Podman Compose.

# Add the Helm repo
helm repo add gateforge \
  https://maximilianopizarro.github.io/gateforge/

# Install
helm install gateforge gateforge/gateforge

Local Development

Run the full stack locally with Podman Compose for development and testing.

# Clone the repo
git clone https://github.com/maximilianoPizarro/gateforge.git

# Build and run
podman-compose up -d --build

Configuration

All parameters are configurable via Helm values: AI model, 3scale credentials, gateway strategy, and more.

helm install gateforge gateforge/gateforge \
  --set ai.apiKey=YOUR_KEY \
  --set threescale.adminApi.url=URL \
  --set connectivityLink.gatewayStrategy=shared

Gateway Strategy Comparison

StrategyDescriptionBest For
sharedAll APIs share a single Istio Gateway with multiple HTTPRoutesSmall to medium clusters, simpler management
dualTwo Gateways (internal/external) with APIs split by exposure levelOrganizations requiring network segmentation
dedicatedOne Gateway per 3scale Product, full isolation per APILarge enterprises, strict isolation requirements

Low Effort

Phase 1 — Multiple 3scale Sources

Support N 3scale Admin API endpoints. Aggregate products from all sources with source-cluster tagging. ~2-3 days.

Medium Effort

Phase 2 — Multi-Cluster Apply

Support multiple kubeconfig entries (as Secrets). Cluster selector in migration wizard. Per-cluster RBAC validation. ~1-2 weeks.

High Effort

Phase 3 — Hub-Spoke Architecture

ACM integration. Centralized migration history. Cross-cluster validation. Federation of audit logs. Longer-term roadmap.

Integration Points

  • Red Hat ACM: GateForge as a managed addon; leverage multi-cluster inventory and kubeconfig distribution
  • ArgoCD multi-cluster: Use ArgoCD cluster Secrets for spoke discovery
  • Kuadrant multi-cluster DNS/TLS: Kuadrant 1.3+ supports multi-cluster policies that work with GateForge-generated resources
  • Fabric8 KubernetesClient: Already supports creating per-cluster clients from kubeconfig files
